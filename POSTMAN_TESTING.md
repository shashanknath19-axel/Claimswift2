# ClaimSwift Postman Testing Guide

This guide explains how to use the generated Postman artifacts for local API testing.

## Files

- Collection:
  - `run-logs/ClaimSwift-Local-Smoke.postman_collection.json` (latest stable export)
- Environment:
  - `run-logs/postman-env-<timestamp>.json` (generated by smoke test)
- Generator script:
  - `run-logs/generate-postman-collection.ps1`

## What is Included

- 86 API requests
- 10 folders grouped by service:
  - Config Server
  - Eureka Server
  - API Gateway
  - Auth Service
  - Claim Service
  - Document Service
  - Assessment Service
  - Payment Service
  - Notification Service
  - Reporting Service

## Import in Postman

1. Open Postman.
2. Click **Import**.
3. Import collection file:
   - `run-logs/ClaimSwift-Local-Smoke.postman_collection.json`
4. Import environment file:
   - `run-logs/postman-env-20260304-132420.json`
5. Select the imported environment in the top-right dropdown.

## Run Order (Recommended)

1. Auth Service
2. Claim Service
3. Document Service
4. Assessment Service
5. Payment Service
6. Notification Service
7. Reporting Service
8. API Gateway / health checks

## Important Variables

Environment already includes:

- Base URLs (`auth_base`, `claim_base`, etc.)
- Tokens (`policy_token`, `adjuster_token`, `manager_token`)
- IDs (`claim_id`, `assessment_id`, `payment_id`, `document_id`, etc.)
- File path for multipart upload (`upload_file`)

If tokens or IDs become invalid, regenerate by running smoke test again:

```powershell
powershell -ExecutionPolicy Bypass -File run-logs\api-smoke-test.ps1
```

Then import the newest `postman-env-*.json`.

## Regenerate Collection

To regenerate a fresh collection from latest smoke report:

```powershell
powershell -ExecutionPolicy Bypass -File run-logs\generate-postman-collection.ps1
```

It will create a new timestamped file:

- `run-logs/ClaimSwift-Local-Smoke-YYYYMMDD-HHMMSS.postman_collection.json`
- and refresh `run-logs/ClaimSwift-Local-Smoke.postman_collection.json`
